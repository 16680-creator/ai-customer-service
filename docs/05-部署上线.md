## 1. Docker 镜像构建

### 1.1 各服务 Dockerfile

在项目根目录创建 `docker/` 目录，为每个服务编写 Dockerfile。

**通用多阶段构建模板**（以 `ai-cs-user` 为例，`docker/ai-cs-user/Dockerfile`）：

```dockerfile
# 构建阶段
FROM eclipse-temurin:17-jdk-alpine AS builder
WORKDIR /app
COPY . .
RUN ./mvnw clean package -pl ai-cs-user -am -DskipTests

# 运行阶段
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app

# 安装必要工具
RUN apk add --no-cache tzdata curl \
    && cp /usr/share/zoneinfo/Asia/Shanghai /etc/localtime \
    && echo "Asia/Shanghai" > /etc/timezone

# 创建非 root 用户
RUN addgroup -S aics && adduser -S aics -G aics

COPY --from=builder /app/ai-cs-user/target/ai-cs-user-1.0.0-SNAPSHOT.jar app.jar

RUN chown -R aics:aics /app
USER aics

EXPOSE 8081

ENTRYPOINT ["java", \
    "-Xms256m", "-Xmx512m", \
    "-XX:+UseG1GC", \
    "-XX:+HeapDumpOnOutOfMemoryError", \
    "-XX:HeapDumpPath=/app/logs/", \
    "-jar", "app.jar"]
```

**各服务 Dockerfile 差异**：

| 服务 | 端口 | JVM 内存 | 特殊依赖 |
|------|------|---------|---------|
| ai-cs-gateway | 8080 | 256m-512m | 无 |
| ai-cs-user | 8081 | 256m-512m | 无 |
| ai-cs-knowledge | 8082 | 256m-512m | 无 |
| ai-cs-chat | 8083 | 512m-1024m | Spring AI 需要更多内存 |
| ai-cs-search | 8084 | 256m-512m | 无 |
| ai-cs-message | 8085 | 256m-512m | 无 |
| ai-cs-notify | 8086 | 256m-512m | 无 |

### 1.2 批量构建脚本

创建 `scripts/build-images.sh`：

```bash
#!/bin/bash

# 项目版本
VERSION=${1:-1.0.0-SNAPSHOT}
REGISTRY=${2:-registry.cn-hangzhou.aliyuncs.com/aics}

# 服务列表
SERVICES=(
    "ai-cs-gateway:8080:256m:512m"
    "ai-cs-user:8081:256m:512m"
    "ai-cs-knowledge:8082:256m:512m"
    "ai-cs-chat:8083:512m:1024m"
    "ai-cs-search:8084:256m:512m"
    "ai-cs-message:8085:256m:512m"
    "ai-cs-notify:8086:256m:512m"
)

echo "========================================="
echo "Building AICS Docker Images"
echo "Version: ${VERSION}"
echo "Registry: ${REGISTRY}"
echo "========================================="

# 先编译所有模块
echo "[1/2] Compiling all modules..."
./mvnw clean package -DskipTests
if [ $? -ne 0 ]; then
    echo "Build failed!"
    exit 1
fi

# 构建镜像
echo "[2/2] Building Docker images..."
for SERVICE_CONFIG in "${SERVICES[@]}"; do
    IFS=':' read -r SERVICE PORT XMS XMX <<< "$SERVICE_CONFIG"

    echo "-------------------------------------------"
    echo "Building ${SERVICE}..."

    docker build \
        -f docker/${SERVICE}/Dockerfile \
        --build-arg SERVICE_NAME=${SERVICE} \
        -t ${REGISTRY}/${SERVICE}:${VERSION} \
        -t ${REGISTRY}/${SERVICE}:latest \
        .

    if [ $? -eq 0 ]; then
        echo "✅ ${SERVICE} built successfully"
    else
        echo "❌ ${SERVICE} build failed"
        exit 1
    fi
done

echo "========================================="
echo "All images built successfully!"
echo "========================================="

# 推送镜像
read -p "Push images to registry? (y/n) " -n 1 -r
echo
if [[ $REPLY =~ ^[Yy]$ ]]; then
    for SERVICE_CONFIG in "${SERVICES[@]}"; do
        IFS=':' read -r SERVICE _ <<< "$SERVICE_CONFIG"
        docker push ${REGISTRY}/${SERVICE}:${VERSION}
        docker push ${REGISTRY}/${SERVICE}:latest
    done
    echo "All images pushed!"
fi
```

使用方式：

```bash
chmod +x scripts/build-images.sh

# 构建镜像（默认版本）
./scripts/build-images.sh

# 指定版本和仓库
./scripts/build-images.sh 1.0.0 registry.cn-hangzhou.aliyuncs.com/aics
```

### 2.2 启动与停止

```bash
# 启动所有服务
docker compose -f docker-compose-all.yml up -d

# 查看服务状态
docker compose -f docker-compose-all.yml ps

# 查看某个服务日志
docker compose -f docker-compose-all.yml logs -f ai-cs-chat

# 重启某个服务
docker compose -f docker-compose-all.yml restart ai-cs-chat

# 停止所有服务
docker compose -f docker-compose-all.yml down
```

## 4. K8s 资源清单说明

### 4.1 资源类型对照

| 资源类型 | 用途 | 适用组件 |
|---------|------|---------|
| **Namespace** | 资源隔离 | aics 命名空间 |
| **StatefulSet** | 有状态服务 | MySQL, Redis, Nacos, Elasticsearch, RocketMQ |
| **Deployment** | 无状态服务 | 所有微服务, Gateway |
| **Service** | 服务发现与负载均衡 | 所有组件 |
| **ConfigMap** | 非敏感配置 | Nacos 地址, 命名空间等 |
| **Secret** | 敏感配置 | 数据库密码, API Key, JWT 密钥 |
| **PersistentVolumeClaim** | 持久化存储 | MySQL, Elasticsearch, MinIO |
| **Ingress** | 外部访问入口 | Gateway |

## 5. Jenkins CI/CD 流水线

### 5.1 Jenkinsfile

创建 `Jenkinsfile` 在项目根目录：

```groovy
pipeline {
    agent any

    environment {
        REGISTRY = 'registry.cn-hangzhou.aliyuncs.com/aics'
        VERSION = "${env.BUILD_NUMBER}"
        NAMESPACE = 'aics'
        KUBECONFIG = credentials('kubeconfig')
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Build') {
            steps {
                sh './mvnw clean package -DskipTests'
            }
        }

        stage('Unit Tests') {
            steps {
                sh './mvnw test'
            }
            post {
                always {
                    junit '**/target/surefire-reports/*.xml'
                }
            }
        }

        stage('Build Images') {
            parallel {
                stage('Gateway Image') {
                    steps {
                        buildAndPush('ai-cs-gateway')
                    }
                }
                stage('User Service Image') {
                    steps {
                        buildAndPush('ai-cs-user')
                    }
                }
                stage('Knowledge Service Image') {
                    steps {
                        buildAndPush('ai-cs-knowledge')
                    }
                }
                stage('Chat Service Image') {
                    steps {
                        buildAndPush('ai-cs-chat')
                    }
                }
                stage('Search Service Image') {
                    steps {
                        buildAndPush('ai-cs-search')
                    }
                }
                stage('Message Service Image') {
                    steps {
                        buildAndPush('ai-cs-message')
                    }
                }
                stage('Notify Service Image') {
                    steps {
                        buildAndPush('ai-cs-notify')
                    }
                }
            }
        }

        stage('Deploy to K8s') {
            steps {
                sh """
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-gateway ai-cs-gateway=\${REGISTRY}/ai-cs-gateway:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-user ai-cs-user=\${REGISTRY}/ai-cs-user:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-knowledge ai-cs-knowledge=\${REGISTRY}/ai-cs-knowledge:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-chat ai-cs-chat=\${REGISTRY}/ai-cs-chat:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-search ai-cs-search=\${REGISTRY}/ai-cs-search:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-message ai-cs-message=\${REGISTRY}/ai-cs-message:\${VERSION} -n \${NAMESPACE}
                    kubectl --kubeconfig=\${KUBECONFIG} set image deployment/ai-cs-notify ai-cs-notify=\${REGISTRY}/ai-cs-notify:\${VERSION} -n \${NAMESPACE}
                """
            }
        }

        stage('Health Check') {
            steps {
                sh """
                    kubectl --kubeconfig=\${KUBECONFIG} rollout status deployment/ai-cs-gateway -n \${NAMESPACE} --timeout=120s
                    kubectl --kubeconfig=\${KUBECONFIG} rollout status deployment/ai-cs-user -n \${NAMESPACE} --timeout=120s
                    kubectl --kubeconfig=\${KUBECONFIG} rollout status deployment/ai-cs-chat -n \${NAMESPACE} --timeout=120s
                """
            }
        }
    }

    post {
        failure {
            echo 'Deployment failed! Rolling back...'
            sh """
                kubectl --kubeconfig=\${KUBECONFIG} rollout undo deployment/ai-cs-gateway -n \${NAMESPACE}
                kubectl --kubeconfig=\${KUBECONFIG} rollout undo deployment/ai-cs-user -n \${NAMESPACE}
                kubectl --kubeconfig=\${KUBECONFIG} rollout undo deployment/ai-cs-chat -n \${NAMESPACE}
            """
        }
        success {
            echo 'Deployment successful!'
        }
    }
}

def buildAndPush(String service) {
    sh """
        docker build -f docker/${service}/Dockerfile -t \${REGISTRY}/${service}:\${VERSION} -t \${REGISTRY}/${service}:latest .
        docker push \${REGISTRY}/${service}:\${VERSION}
        docker push \${REGISTRY}/${service}:latest
    """
}
```

## 7. 常见部署问题排查

### 7.1 服务启动失败

```bash
# 查看 Pod 状态
kubectl get pods -n aics

# 查看 Pod 事件
kubectl describe pod <pod-name> -n aics

# 查看容器日志
kubectl logs <pod-name> -n aics
kubectl logs <pod-name> -n aics --previous  # 查看上次崩溃的日志

# 常见原因：
# 1. ImagePullBackOff → 检查镜像地址和凭证
# 2. CrashLoopBackOff → 检查应用日志，通常是配置错误
# 3. OOMKilled → 增加 memory limit
```

### 7.2 服务间无法通信

```bash
# 检查 Service 是否存在
kubectl get svc -n aics

# 检查 Endpoints
kubectl get endpoints -n aics

# 在 Pod 内测试连通性
kubectl exec -it <pod-name> -n aics -- sh
curl http://ai-cs-user:8081/actuator/health
curl http://nacos:8848/nacos/v1/ns/service/list?pageNo=1&pageSize=10

# 检查 DNS 解析
kubectl exec -it <pod-name> -n aics -- nslookup nacos
```

### 7.3 Nacos 注册失败

```bash
# 检查 Nacos 服务列表
curl "http://<nacos-ip>:8848/nacos/v1/ns/service/list?pageNo=1&pageSize=100&namespaceId=aics"

# 常见原因：
# 1. namespace 配置错误 → 确认 namespace ID 为 aics
# 2. 网络不通 → 检查 Service 和网络策略
# 3. Nacos 未就绪 → 增加启动等待时间
```

### 7.4 数据库连接失败

```bash
# 在 Pod 内测试数据库连接
kubectl exec -it <pod-name> -n aics -- sh
mysql -h mysql-master -P 3306 -u root -p

# 检查 Secret 中的密码
kubectl get secret aics-secrets -n aics -o jsonpath='{.data.mysql-password}' | base64 -d

# 常见原因：
# 1. 密码错误 → 检查 Secret 配置
# 2. MySQL 未就绪 → 增加 initialDelaySeconds
# 3. 数据库不存在 → 检查 init SQL 是否执行
```

### 7.5 AI 服务调用超时

```bash
# 检查 Ollama / OpenAI 连通性
kubectl exec -it <chat-pod> -n aics -- sh
curl http://ollama:11434/api/tags  # Ollama
curl https://api.openai.com/v1/models -H "Authorization: Bearer $OPENAI_API_KEY"  # OpenAI

# 常见原因：
# 1. API Key 无效 → 检查 Secret 配置
# 2. 网络不通 → 检查网络策略和出口规则
# 3. 模型未下载 → 执行 ollama pull qwen2.5:7b
# 4. 内存不足 → 增加 ai-cs-chat 的 memory limit
```
